Per-Workspace API Rate Limiting

Feature 008 · status: approved

Why

Workspaces share gateway infrastructure. A single high-volume tenant can currently exhaust capacity and degrade latency for every other workspace. We need per-tenant isolation at the edge.

User stories

  • As an operator, I want per-workspace limits so one tenant can't starve others.
  • As a client developer, I want clear 429 + Retry-After responses so I can back off correctly.

Acceptance criteria

  • Requests past the limit return 429 with Retry-After.
  • Limits are configurable per route, default 1000 / min. added
  • Remaining quota is surfaced via X-RateLimit-Remaining.

Out of scope

Cross-region token-bucket fairness — tracked separately in #151.
specs/008-rate-limiting/plan.md +47

Implementation Plan

Architecture

A RateLimiter backed by Redis INCR with a per-key TTL window. Enforcement happens in gateway middleware before the route handler runs.

Components

  • limits.ts — config resolution (defaults + per-route overrides)
  • rateLimiter.ts — counting & allow/deny verdict
  • middleware.ts — enforcement + response headers

Data model

key:  rl:{workspaceId}:{route}
value: counter (INCR)
ttl:  cfg.windowSec

Risks

Redis latency on the hot path. Mitigation: pipeline INCR+EXPIRE; circuit-break to fail-open if Redis is unavailable (follow-up).
